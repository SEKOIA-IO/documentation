# Event Externalisation

When a detection rule matches suspicious activity in your environment, Sekoia.io generates an **alert** and simultaneously attaches the relevant **events** (raw logs) to it. This process of permanently storing and indexing those events so they remain accessible within the alert is called **event externalisation**.

In short: event externalisation is what allows you, as an analyst, to **see the supporting evidence** directly inside an alert, enabling faster and more accurate investigation.


## How It Works

When an alert is triggered, two mechanisms run **in parallel**:

| Step | What Happens |
|---|---|
| **1. Alert generation** | The platform detects a rule match and creates the alert record |
| **2. Event externalisation** | The relevant events are fetched, permanently indexed, and attached to the alert |

!!! note
    Because these two steps run in parallel, there can be a **short latency** between when the alert appears in your queue and when all associated events are fully visible within it.


## What This Means for Your Investigations

### Events Are Preserved

Once externalised, the events linked to an alert are **durably stored** in a dedicated index, separate from the raw event stream. This means:

- Events remain accessible even after the standard retention period of the raw events index.
- You can always go back to an alert and review its original evidence.

### The Bell Icon on Events

On the Events page or within an alert, you may notice a **bell icon** on certain events. This means:

- The event is involved in the **current alert AND at least one other alert**.
- If the event is *only* involved in the current alert, the bell icon is **not** displayed.

This helps you identify events that are part of a broader attack pattern across multiple alerts.

## Understanding Timestamps

Understanding timestamp fields is key to accurate investigation and timeline reconstruction:

| Field | Meaning |
|---|---|
| **Alert date** | Date of the **first trigger** â€” it does not update even if new events enrich the alert later |
| **`timestamp`** | When the event was **generated by the source system** (in UTC) |
| **`event.created`** | When **Sekoia.io acknowledged/ingested** the event |
| **`event.start`** | When the monitored activity **began** |
| **`event.end`** | When the monitored activity **ended** |

The general ordering rule is:

```
event.start â‰¤ event.end â‰¤ event.created
```

!!! info
    Events must be sent to Sekoia.io in UTC. If an event is sent in local time (e.g. UTC+2) with a 09:00 timestamp, it will be stored as 09:00 UTC.

## Why Are Events Missing or Delayed in an Alert?

If you notice that an alert contains fewer events than expected, or that events appear with a delay, here are the most common explanations:

**Externalisation is still in progress**

The pipeline is running; events will appear shortly. Refresh the alert after a few seconds.

**The alert was triggered with a delay**

This can happen when:

- A new IOC is published and old events are retrospectively scanned â€” if a match is found, the rule will automatically trigger an alert.
- Logs from the source reached Sekoia.io with a delay, for example:
    - The log collector was temporarily interrupted and buffered logs locally before resending.
    - Old logs were reingested manually.

**The alert date reflects the first trigger only**

Even if new events are added later to enrich the alert, the alert date stays anchored to the first trigger.

!!! warning
    Do not rely solely on the alert date for timeline reconstruction. Always use the `event.start` and `timestamp` fields for accurate sequencing.


## Tips for Analysts

!!! tip
    - **Refresh the alert** if events are not yet visible â€” externalisation can take a few seconds.
    - **Check `timestamp` vs. `event.created`** to understand whether logs arrived late to the platform.
    - **Pay attention to the bell icon** â€” it signals that an event is shared with another alert, which may indicate a wider incident.
    - **Use `event.start` and `timestamp`** for accurate timeline reconstruction rather than the alert date.
